## Versioning — 4 locations must stay in sync

When bumping the package version, **all four locations must be updated together** or the PyPI release will be broken:

| File | Field |
|------|-------|
| `pyproject.toml` | `version = "X.Y.Z"` under `[project]` |
| `rust/periodica_core/Cargo.toml` | `version = "X.Y.Z"` under `[package]` |
| `src/periodica/__init__.py` | `__version__ = "X.Y.Z"` |
| Git tag | `vX.Y.Z` pushed to origin |

After committing and tagging, create a **GitHub Release** from the tag to trigger the PyPI publish workflow (`release: published` event). The workflow does **not** fire on a tag push alone.

> **Important:** `pyproject.toml version` is what PyPI indexes — not the git tag. Pre-release suffixes like `2.0.0a0` are **not** matched by `>=2.0.0` in pip. Always use clean `X.Y.Z` for stable releases.

# Periodica

## Project Philosophy & Senior Development Standards

These reflect senior-level software engineering discipline: clarity,
maintainability, extensibility, automation, and long-term sustainability.

### 1. Core Architectural Principles (SOLID + Beyond)

Apply SOLID principles rigorously:
- **Single Responsibility:** each class / module has one clear reason to change.
- **Open / Closed:** entities are open for extension but closed for modification.
- **Liskov Substitution:** subtypes must be fully substitutable for base types.
- **Interface Segregation:** prefer small, client-specific interfaces over large ones.
- **Dependency Inversion:** depend on abstractions, not concretions.

**DRY + Strategic Generalization:** aggressively eliminate duplication. When
similar logic appears across modules or projects, extract it into well-designed
base classes, abstract base classes, or shared libraries using proper
inheritance and composition. Favor composition over inheritance when behaviour
is likely to vary.

**Separation of Concerns:** keep UI, business logic, data access, and
configuration strictly separated.

### 2. Python-Specific Standards

- **UI Layer:** every Python project with a graphical interface must use
  KivyMD (Material Design for Kivy) for consistency, modern look, and
  cross-platform capability.
- **Build & Distribution:**
  - Every Python project must include a `build.bat` (Windows) and an
    equivalent shell script for other platforms.
  - Target ≥ 90 % code coverage on unit + integration tests before release.
  - Include automated linting (ruff / black), type checking (mypy or pyright),
    and tests as part of the build pipeline.

### 3. Documentation Standards

- Every project must maintain up-to-date HTML documentation focused on
  "How To" basics (setup, common tasks, architecture overview, key workflows).
- Keep the documentation in the repository root or `/docs` folder and
  version-controlled.

### 4. Additional Senior Engineering Practices

**Code Quality & Maintainability:**
- Clear, self-documenting code with meaningful names.
- Comprehensive but focused unit and integration tests.
- Logging and error handling that aids debugging without verbosity.
- Consistent naming conventions and project structure across repositories.

**Automation First:**
- Automate builds, tests, packaging, and documentation generation.
- Minimise manual steps required to go from clone → run → deploy.
